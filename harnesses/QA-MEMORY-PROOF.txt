Phase 1 — Backend CRUD + grep + tag filter + usage tracking + warm invalidation
-------------------------------------------------------------------------------

Created three memories with different sources:
  $ curl -X POST .../memory -d '{"text":"For UI changes...","tags":["ui","antd"],"priority":2}'
    → id=5738b9a0..., source=ui
  $ curl -X POST .../memory -d '{"text":"PR titles must...","source":"slack","source_user_id":"U02ABC123"}'
    → id=b650f545..., source=slack
  $ curl -X POST .../memory -d '{"text":"Never write LiteLLM tokens to localStorage...","type":"constraint","source":"agent"}'
    → id=fa8f8b56..., source=agent

After one list:
              memory_id               | priority | times_applied |     last_applied_at
--------------------------------------+----------+---------------+-------------------------

→ All three returned. times_applied bumped to 1 (matching).
→ Ordering: priority DESC, times_applied DESC, created_at DESC (correct).

Grep:
  $ curl ".../memory?q=Tag"     → 1 hit:  For UI changes always use shadcn Tag...
  $ curl ".../memory?q=SHADCN"  → 1 hit:  For UI changes always use shadcn Tag... (case-insensitive)

Tag filter:
  $ curl ".../memory?tag=ui"  → 2 hits (the UI rule + the localStorage constraint, both tagged ui)
  $ curl ".../memory?tag=git" → 1 hit  (PR titles)

After all the grep + tag-filter calls, the counts diverge — only the rows actually returned bump:
 priority | times_applied | substring (first 50 chars)
----------+---------------+----------------------------------------------------
        2 |             4 | For UI changes always use shadcn Tag, never @tremo
        1 |             2 | PR titles must start with fix:, feat:, or chore:.
        1 |             2 | Never write LiteLLM tokens to localStorage — use s

PATCH disable:
  $ curl -X PATCH .../memory/5738b9a0... -d '{"disabled":true}'
    → disabled=True
  $ curl .../memory  → 2 active memories (the disabled one is hidden) ✓

PATCH priority bump:
  $ curl -X PATCH .../memory/fa8f8b56... -d '{"priority":5}'
    → priority=5 ✓

Warm-task invalidation:
  Inserted a fake warm row for the agent:
    INSERT INTO managed_agent_warm_task (..., status) VALUES (..., 'warm');
  Then saved a new memory:

  Verified the warm task is now dead with the right reason:
    status | failure_reason
    --------+----------------
     dead   | memory_changed   ← invalidateWarmTasks() fired correctly ✓

DELETE returned 204; final active count was 2 after deleting one.

Phase 3 — Claude actually calls the tools (the real validation)
---------------------------------------------------------------

Booted the claude-agent-sdk harness pointed at the LiteLLM gateway with
AGENT_ID + LAP_BASE_URL + LAP_AUTH_TOKEN set, and the same system prompt
shape the platform would inject in production.

WRITE — sent a message containing the explicit cue:

  POST /session/<harness_session_id>/message
  body: {"parts":[{"type":"text","text":"remember: Always run black before committing."}]}

Before: 3 memory rows for this agent.
After:  4 memory rows.

The new row:
  id        = 3920ebbe-5492-4b31-b7ae-6fcbf1e85878
  source    = agent              ← came via the tool, NOT the ui/slack paths
  text      = "Always run black before committing code changes."
              (Claude generalized "remember:" into a clean rule)
  priority  = 4 (Claude picked it)

→ Confirms: Claude saw the system-prompt rules, recognized "remember:" as
  a durable cue, invoked save_memory via MCP, the in-process tool POSTed
  to LAP, LAP persisted the row with source=agent.

READ — separate session, asked it to search:

  POST /session/<new_session>/message
  body: "Use the search_memory tool to find memories related to PR titles..."

times_applied snapshot:

  BEFORE:
   priority | times_applied
   ---------+---------------
          5 |             7
          4 |             0   ← the "Always run black" lesson, never matched yet
          1 |             7   ← "PR titles must start with..."

  AFTER (just one tool call later):
   priority | times_applied
   ---------+---------------
          5 |             7   ← unchanged: not returned by this query
          4 |             0   ← unchanged: not returned
          1 |             8   ← bumped: this is the only row matching "PR titles"

→ Confirms: Claude called search_memory with a relevant query, only the
  matching row's times_applied incremented, server-side bumpUsage works
  correctly, and the agent received the row to reason over.
